# Reader Status Policy

This policy defines how README reader labels move from experimental to mature.
It separates translation completeness from tested behavior: a backend can be
translated and still remain experimental until fixture parity, benchmark RSS,
CI, and dependency risks are covered.

## Status Labels

| Label | Meaning | README language |
| --- | --- | --- |
| Experimental | Blind translation, synthetic-only tests, missing representative data, or known parity drift without an accepted reason. | `Experimental` |
| Fixture-verified | One or more representative fixtures open and match reference OpenSlide metadata and sampled pixel parity in automation. | `Fixture-verified (<covered subset>)` |
| Conditionally mature | Exact fixture parity, benchmark/RSS baseline, known unsupported features documented, and CI coverage exists, but fixture breadth or platform coverage is incomplete. | `Conditionally mature` |
| Mature | Broad public/private fixture matrix, exact or policy-accepted pixel behavior, RSS/speed thresholds, CI gates, and packaging/dependency checks are all in place. | `Mature` |

## Promotion Gates

Promotion is per reader, not global.

1. The fixture must be listed in `fixtures/manifest.toml`.
   Public fixtures must have a reproducible
   `scripts/download-openslide-testdata.py` command with the required selector,
   extraction, and distributable-license flags.
   Public manifest path tokens must be covered by that downloader selection.
   Private fixture paths must point under `/big/henriksson/ome_images`, and
   missing fixtures must keep an empty path with a missing-status row.
2. Expected parity must be listed in `fixtures/expected-parity.toml`.
   Pixel parity reports must use the expected sampling parameters.
   Exact and known-drift expectations must record Rust/reference checksum
   evidence. Exact rows must have equal Rust/reference checksum pairs; known
   drifts must include at least one differing pair.
   Exact and known-drift parity reports must carry metadata evidence for
   vendor, level count, dimensions, and downsamples. Pixel-enabled exact and
   known-drift parity reports must also carry sampled `pixel_stats` evidence
   and associated-image name/dimension evidence.
3. Benchmark and RSS baselines must be listed in `fixtures/bench-baseline.json`
   or regenerated by a documented CI/job artifact. Benchmark reports must use
   the baseline sampling parameters.
   Exact and known-drift rows must record Rust/reference read time, RSS, speed
   ratio, and RSS ratio; exact-limited rows must record numeric ranges.
   Speed/RSS ratios must match the recorded Rust/reference measurements, and
   limited ranges must be positive and ordered.
   Generated reports for measured fixtures must include Rust/reference read
   payloads, checksums, region/pixel counts, and RSS values even when strict
   regression thresholds are disabled.
4. Per-level diagnostic reports must use the sampling parameters recorded in
   `fixtures/level-baseline.json` and include per-sample level coordinates,
   level-0 coordinates, region sizes, and checksums.
5. CI must run at least metadata parity for a small public fixture. Pixel
   parity may be non-blocking only when the expected baseline records a known
   codec drift or reference-open limitation.
   Parity smoke PR triggers must include maturity docs and audit files, not
   only source changes.
6. Unsupported compression, storage layout, multi-plane, or associated-image
   behavior must be described in README notes and in the fixture expectation.
7. Package verification must confirm that native helper sources required by
   `build.rs` are included in the crate package.
8. `scripts/check-audit-baselines.py` must pass for the checked-in fixture,
   expected-parity, benchmark baseline, level baseline, and fixture matrix
   files.
   Each checked-in audit contract file must declare `schema_version = 1`.
9. Any reader labelled `Fixture-verified` must have each listed evidence
   fixture represented by a `covered` case in `fixtures/matrix.toml`.
10. Every reader-status evidence or blocker fixture must appear in the reader's
   fixture matrix; blockers must be represented by a non-`covered` case.
   Non-`covered` matrix fixtures must also be listed as reader blockers, so
   unsupported layouts and missing data stay visible at the README status
   boundary.
11. Every expected-parity fixture must appear in the fixture matrix, and every
    matrix fixture must have an expected-parity row.
    Matrix case status must agree with the expected-parity status family for
    each referenced fixture. `pending` matrix cases may have no fixture
    references yet, but must explain the missing fixture, policy, or codec work
    needed before promotion.
12. Manifest, expected-parity, benchmark, level-baseline, and matrix entries
    must agree on the fixture's reader id.
    Benchmark and level-baseline statuses must also agree with expected-parity
    statuses.
13. README reader statuses must use the policy label families. A
    `Fixture-verified` status must name its covered subset, `Experimental`
    statuses must list blockers, and `Feature-supported` statuses must list
    both evidence and blockers.
    The README benchmark snapshot must include one row for every reader-status
    entry, and its metric columns must agree with whether the reader has
    measured benchmark baselines or only blocked/missing rows.
14. `Fixture-verified` evidence fixtures must be exact in both the manifest and
    expected-parity files. Blocker fixtures must not be exact in either file.
15. `Conditionally mature` and `Mature` statuses must have no blocker fixtures,
    exact evidence fixtures, and exact benchmark evidence. `Conditionally
    mature` requires at least two covered matrix cases; `Mature` requires at
    least three. Neither status may have non-covered matrix cases. Both labels
    require the strict benchmark runner profile from
    `fixtures/bench-baseline.json` to be `active` in
    `fixtures/runner-status.toml`.

## Current Reader State

Current reader labels, evidence fixtures, blocker fixtures, benchmark evidence,
matrix coverage, and next promotion blockers are generated in
`docs/maturity-report.md` from the checked-in audit contracts. Do not maintain a
second hand-written promotion table here; stale duplicated status summaries
hide real drift. Refresh the report with:

```sh
python3 scripts/maturity-report.py --output docs/maturity-report.md
```

## Benchmark Policy

Benchmarks are comparative guardrails, not marketing claims. A promoted reader
must record read time and maximum RSS against the same reference stack used for
parity. Benchmark jobs publish artifacts on unstable runners and enforce saved
thresholds on stable runners. Strict enforcement is automatic when the
benchmark report `runner_profile` matches
`fixtures/bench-baseline.json.enforcement_policy.strict_runner_profile`; the
nightly workflow exposes this as a manual `runner_profile` input for stable
benchmark runners.

Strict runner readiness is tracked separately from reader parity in
`fixtures/runner-status.toml` and surfaced in `docs/maturity-report.md`.
`external-pending` means the repository-side contract is present but strict
speed/RSS baselines still need a validated self-hosted runner artifact. A
runner may be marked `active` only after `stable-runner-preflight.json` and
`bench-stable.json` validate, at which point `scripts/update-runner-status.py`
writes `last_validated_utc`.

Default thresholds once a stable runner exists:

| Metric | Warning | Failure |
| --- | ---: | ---: |
| Read time regression vs baseline | `> 15%` | `> 25%` |
| RSS regression vs baseline | `> 10%` | `> 20%` |
| Parity status regression | any drift from `exact` | any hard metadata or pixel regression |

## Dependency Policy

README and crate metadata must not describe the crate as pure Rust while
`build.rs` compiles C helpers and links native libraries. The package include
list must contain every file used by `build.rs`; CI should run `cargo package`
to catch missing native helper sources before release.
`scripts/check-audit-baselines.py` enforces the package include list, native
dependency README wording, and CI package smoke contract.

Codec backend and unsupported-layout policy lives in `docs/codec-policy.md`.
The audit validator requires the key JPEG 2000, JPEG XR, libtiff-only layout,
native-helper, and `UnsupportedFormat` decisions to remain documented.

Cache, memory, and error policy lives in `docs/memory-error-policy.md`. The
audit validator requires the shared `TileCache` budget, strict benchmark RSS
gate, translated-reader open failures, and `UnsupportedFormat` rules to remain
documented.
